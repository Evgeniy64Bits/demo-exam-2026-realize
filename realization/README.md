# Развёртывание стенда (реализация)

## Введение

Для начала определимся с текущей задачей.

Нам нужно создать 6 виртуальных машин:

* ISP
* HQ-RTR
* BR-RTR
* HQ-SRV
* HQ-CLI
* BR-SRV

И настроить корректную логику сетевых адаптеров, чтобы между ними была нормальная взаимосвязь.

---

## Способы реализации

Для домашней реализации есть 3 варианта виртуализации:

### 1. Виртуальные машины (VirtualBox / VMware Workstation)

Самый простой способ. Подходит для старта, не требует сложной настройки.

---

### 2. Proxmox VE в виде виртуальной машины

Proxmox устанавливается как виртуальная машина (например, через VirtualBox или VMware).

Принцип работы в целом похож на вариант с установкой на железо.

Плюсы:

* всё находится на одном компьютере
* ближе к реальному экзамену

Минусы:

* сложнее в реализации
* требует больше ресурсов

# 🧾 Заключение по проблеме запуска EcoRouter

## 📌 Исходные условия

* Гипервизор: VMware Workstation Pro
* Внутри: Proxmox VE (nested virtualization)
* Проблемная ВМ: EcoRouter
* CPU: Intel Core i5-11400H

---

## ⚠️ Симптом

* EcoRouter зависает на этапе:

kexec_core: Starting new kernel
* Дальнейшая загрузка отсутствует

---

## 🔍 Проведённая диагностика

Проверено и исключено:

* ✔️ Nested virtualization включён и работает
* ✔️ CPU type: host
* ✔️ Hyper-V отключён
* ✔️ Конфигурации ВМ идентичны рабочим стендам
* ✔️ Проверены CPU flags (включение/исключение)
* ✔️ Использованы hidden=1, q35
* ✔️ Исключён invtsc
* ✔️ Проверены args и чистый конфиг

Дополнительно:

* На других CPU (Intel Core i5-12400F, AMD Ryzen 3 3200G) стенд работает
* В том же VMware Workstation Pro без Proxmox — EcoRouter запускается

---

## 🧠 Причина

👉 Ограничение вложенной виртуализации:

VMware Workstation Pro → Proxmox VE → EcoRouter

В данной конфигурации:

* VMware не пробрасывает необходимые MSR/VMX-функции CPU
* EcoRouter использует kexec (перезапуск ядра)
* второе ядро не может корректно инициализировать CPU

👉 В результате возникает зависание на этапе kexec

---

## ❗️ Важное уточнение

Проблема:

* ❌ не связана с конфигурацией ВМ
* ❌ не связана с Proxmox
* ❌ не связана с самим EcoRouter

👉 Это ограничение конкретной связки:

* nested virtualization
* мобильный CPU Intel Core i5-11400H
* реализация виртуализации в VMware

---

## ✅ Вывод

👉 Запуск EcoRouter в данной архитектуре невозможен (или крайне нестабилен)

### 3. Proxmox VE на реальном железе

Установка на отдельный компьютер.

Схема работы:

* один компьютер — сервер с Proxmox и виртуальными машинами
* второй компьютер — для подключения и настройки через веб-интерфейс

Это максимально приближено к реальному демоэкзамену.

---

### Важные моменты по Proxmox

Для установки потребуется отдельный диск.

Если у вас на компьютере уже стоит Windows и вы не хотите её сносить, то:

* Windows не видит файловую систему Linux (ext4)
* Linux не работает с NTFS как с основной системой

В итоге:

* диск полностью форматируется под ext4
* получается dual boot

Поэтому для меня этот 3 вариант не подходит и использовать его я точно не буду.

---

Также:
для нормальной работы Proxmox желательно проводное подключение.

По информации из сети, через Wi-Fi возможны проблемы с виртуальными коммутаторами и стабильностью стенда.

## Мой подход

Поскольку на экзамене времени даётся немного, я хочу:

* набить руку в быстром конфигурировании
* делать всё по памяти
* параллельно заполнять отчёт

Поэтому я и делаю этот репозиторий:

* как удобную шпаргалку для себя
* и как помощь для других

Потому что хранить всё в txt-файлах на флешке неудобно:

* плохое форматирование
* сложно читать
* сложно структурировать

Поскольку реальный экзамен проходит в среде Proxmox, работа с ним выглядит следующим образом:

мы просто открываем браузер, вводим
[http://ip-сервера:порт]
и выполняем все задания через веб-интерфейс.

Соответственно, 2 и 3 способы (использование Proxmox) являются наиболее приближенными к реальным условиям экзамена.

С веб-интерфейсом Proxmox я уже знаком и работал с ним во время учебной практики в колледже, где мы также решали демоэкзамен.

---

В моём варианте реализации:

* либо я просто продолжаю выполнение модуля 1
* либо создаю отдельные виртуальные машины, где модуль 1 уже настроен, и использую их как базу для модуля 2

---

При повторении конфигурирования это удобно:

* можно делать снимки (snapshots) виртуальных машин
* можно выгружать ВМ вместе с конфигурацией в отдельный каталог

Таким образом:
не нужно каждый раз поднимать всё с нуля, можно быстро вернуться к нужному этапу и продолжить выполнение.

## Выбранный способ

Я буду показывать реализацию через виртуальные машины в VMware Workstation.

Это:

* быстрее для старта
* проще
* знакомо лично мне
* требует меньше ресурсов

---

## Почему VMware, а не VirtualBox

1. По моим наблюдениям, виртуальные машины работают быстрее
2. С 11 ноября 2024 года VMware Workstation стал бесплатным
3. У меня есть образ EcoRouterOS в формате VMDK

VirtualBox такой формат напрямую не использует, поэтому выбор очевиден.

---

## Важно про маршрутизаторы

На демоэкзамене могут использоваться:

* EcoRouterOS
* ALT JeOS

В моём случае:
я буду использовать **EcoRouterOS** для маршрутизаторов HQ-RTR и BR-RTR.

## Характеристики виртуальных машин

* RAM: 1 GB

* CPU: минимум 2 ядра (лучше 4)

  * в HQ-сеть

* CPU: минимум 2 ядра (лучше 4)
* RAM: от 4 GB
* Диск: используется готовый образ
* Сетевые адаптеры:

  * к ISP
  * в BR-сеть

---

### HQ-SRV (ALT Server 10.1)

* CPU: 1 ядро
* RAM: 1–2 GB
* Диск: 30-40 GB
* Сеть: 1 адаптер

---

### HQ-CLI (ALT Workstation 10.1)

* CPU: 2 ядра
* RAM: 2 GB
* Диск: 30-40 GB
* Сеть: 1 адаптер

---

### BR-SRV (ALT Server 10.1)

* CPU: 1 ядро
* RAM: 1–2 GB

### Шаг 21 (НЕОБЯЗАТЕЛЬНО)

Данный шаг является необязательным, но упростит работу с HQ-RTR и BR-RTR как и в среде VMware, так и в Proxmox.

По умолчанию, спустя некоторое время бездействия, EcoRouter разлогинивает вас с учётной записи, что заставляет вас заново вводить логин и пароль

Это поведение можно изменить, прописав данные настройки в линиях консолей console 0 и vty 0 39

```
conf t
line con 0
exec-timeout 0 0
exit
line vty 0 39
exec-timeout 0 0
exit
write memory
```

![Шаг 21](../pictures-r/step21-routers.png)

После пишем exit, перезаходим для применения тайм-аутов

write memory - команда для записи изменений в маршрутизатор. 

Для большинства команд используются сокращённые версии (en, conf t, show run, wr mem, ip a и т.д.)
